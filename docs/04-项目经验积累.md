# 项目经验积累

本文沉淀当前项目内可复用的经验，包括技术难点、踩坑记录、排查路径和已验证解决方案。它不维护通用开发规范，也不记录没有复用价值的一次性流水账。

## 1. 技术难点

| 难点 | 背景 | 当前方案 | 后续注意 |
|---|---|---|---|
| Codex 额度读取不依赖桌面窗口 | 用户关闭 Codex 后仍希望刷新额度 | 启动 `codex.exe app-server --listen stdio://` 并调用 JSON-RPC | 登录态、网络和 CLI 路径仍会影响读取 |
| 小窗和大窗状态一致性 | 重启、热更新、首次安装、模式切换或刷新失败时，小窗可能没有及时拿到数据 | 后端保存 `last-quota.json`，前端维护最后一次可用额度快照，小窗和大窗都从同一展示快照渲染 | UI 状态修改不能让 loading 覆盖已有有效额度 |
| 切换账号后的失败态展示 | 用户切换 Codex 账号或登录态失效后，继续展示旧缓存会误导用户 | 刷新失败时进入“暂时无法获取”状态并清空当前展示额度，只保留缓存供下次启动或成功前兜底 | 失败态不能冒充当前账号数据，错误原因需尽量归类为网络、登录、超时或无额度数据 |
| 不同机器 Codex 安装路径不同 | 其他电脑的 Codex CLI 可能位于 hash 子目录 | 自动查找 direct path 和 `bin\<目录>\codex.exe` | 未来 Codex 安装结构变化时继续扩展查找逻辑 |
| Windows 编码差异 | PowerShell 5 对无 BOM 中文脚本解析不稳定，Tauri 又不接受带 BOM JSON | 脚本提示尽量 ASCII，写出的 JSON 保持 UTF-8 无 BOM | 修改版本脚本后必须验证 Tauri 配置可解析 |

## 2. 踩坑记录

### 小窗卡在“正在读取”

| 项 | 内容 |
|---|---|
| 现象 | 大窗已经能显示额度，小窗仍显示 `Codex: 正在读取` |
| 影响 | 用户以为小窗无法读取额度，尤其在另一台电脑首次安装时明显 |
| 原因 | 前端曾把“刷新状态”和“可展示数据”绑定在同一个 `quota/status` 组合上；模式切换、刷新失败、缓存水合或事件顺序异常时，小窗可能看到空 `quota`，即使大窗或缓存已经有可用额度 |
| 解决方案 | 增加 `lastGoodQuota` 和 `displayQuota`；实时读取、后端缓存和 localStorage 读取成功后统一写入最后可用快照；小窗和大窗都从 `displayQuota` 渲染，loading 只表示后台正在请求，不再覆盖已有额度 |
| 验证方式 | `npm run typecheck`；涉及 Rust 后端时再跑 `cargo check`；发布前用 `scripts/verify-release.ps1` 和便携版实测小窗/大窗切换 |
| 后续注意 | 修改前端 loading、缓存、刷新失败处理或模式切换逻辑时，必须验证小窗和大窗在成功、失败、缓存存在、缓存为空四类场景下表现一致 |

### 切换账号后仍显示旧信息

| 项 | 内容 |
|---|---|
| 现象 | 切换 Codex 账号、网络断开或登录态失效后，刷新失败时界面可能继续显示旧账号额度或旧计划信息 |
| 影响 | 用户会误以为当前账号额度已刷新成功 |
| 原因 | 前端曾在刷新失败时优先回落到 `lastGoodQuota/displayQuota`，这适合短暂失败兜底，但不适合账号切换和登录状态变化 |
| 解决方案 | 刷新失败时设置 `status=error`、清空当前 `quota`，展示“暂时无法获取”和具体失败原因；刷新成功时用后端实时结果覆盖展示快照和 localStorage；启动水合优先使用后端缓存，localStorage 只兜底 |
| 验证方式 | `npm run typecheck`、`cargo check`；真实 Codex app-server 读取验证；必要时断网或退出登录观察失败态 |
| 后续注意 | 修改失败态和缓存水合时必须区分“后端缓存”“WebView localStorage”和“刷新失败后的当前展示”，不能让旧账号 localStorage 覆盖实时读取结果 |

### 其他电脑无法获取数据

| 项 | 内容 |
|---|---|
| 现象 | 本机正常，其他电脑安装后无法读取额度 |
| 影响 | 发布版在不同机器上不可用 |
| 原因 | 工具只查找 `%LOCALAPPDATA%\OpenAI\Codex\bin\codex.exe`，但其他电脑 Codex CLI 位于 `bin\<版本或哈希目录>\codex.exe` |
| 解决方案 | 在 `quota.rs` 中增加 `find_nested_codex_exe`，从 `bin` 子目录中按修改时间优先选择 `codex.exe` |
| 验证方式 | 2.0.1 发布包校验，包含 Codex app-server 额度读取 |
| 后续注意 | 报“无法启动 app-server”时优先收集真实 `codex.exe` 路径 |

### 版本脚本写出 BOM 导致构建失败

| 项 | 内容 |
|---|---|
| 现象 | 版本同步后 Tauri 读取 `tauri.conf.json` 失败 |
| 影响 | `cargo check` 或打包被 JSON 解析错误打断 |
| 原因 | PowerShell 脚本把 JSON 写成 UTF-8 BOM，Tauri JSON 解析器不接受 |
| 解决方案 | 版本脚本统一用 `System.Text.UTF8Encoding($false)` 写 UTF-8 无 BOM |
| 验证方式 | 重新保存版本文件后运行 `cargo check` |
| 后续注意 | PowerShell 脚本本身如果无 BOM 且含中文，又可能被 Windows PowerShell 5 按 ANSI 解析，因此脚本提示文本尽量 ASCII |

### 窗口宽度调整暴露数据链路问题

| 项 | 内容 |
|---|---|
| 现象 | 修改大窗宽度并重启开发版后，小窗/大窗数据显示不一致 |
| 影响 | 看起来像尺寸改动导致额度读取坏掉 |
| 原因 | 宽度不是根因，重启触发了原本不稳定的前端初始化和缓存路径 |
| 解决方案 | 后端保存最后一次成功额度，前端启动先读后端缓存 |
| 验证方式 | `npm run typecheck`、`cargo check`、重启开发版观察状态文件和小窗/大窗 |
| 后续注意 | 尺寸、样式、窗口状态修改也要关注数据初始化路径 |

## 3. 排查路径

| 问题类型 | 排查顺序 | 常用命令或入口 |
|---|---|---|
| 读取不到额度 | Codex 是否安装登录 -> `codex.exe` 路径 -> app-server 是否能启动 -> JSON-RPC 是否返回 rate limits | `where.exe codex`、搜索 `%LOCALAPPDATA%\OpenAI\Codex\bin`、`scripts/verify-release.ps1` |
| 暂时无法获取 | 网络是否可用 -> Codex 是否已登录 -> 登录态是否过期 -> app-server 是否超时或返回错误 | 查看界面失败原因、手动刷新、托盘重启、`scripts/verify-release.ps1` |
| 小窗显示正在读取 | 后端缓存是否存在 -> 前端 `lastGoodQuota/displayQuota` 是否有值 -> loading 是否覆盖数据 -> 实时刷新是否失败 | 查看 `last-quota.json`、检查 `App.svelte` |
| 打包或 check 失败 | 版本号是否一致 -> JSON 是否带 BOM -> Node/Rust 环境是否可用 -> 构建产物是否存在 | `npm run typecheck`、`cargo check`、`scripts/verify-release.ps1` |
| 发布文件混乱 | 根目录 EXE 版本 -> README 下载说明 -> `.gitignore` 放行规则 -> Git 暂存清单 | `git status --short`、`git diff --cached --name-only` |

## 4. 已验证解决方案

| 场景 | 方案 | 验证方式 | 适用边界 |
|---|---|---|---|
| Codex 窗口关闭后继续刷新 | 自行启动 `codex.exe app-server --listen stdio://` | 实际读取额度成功 | Codex 已安装、登录态有效、网络可用 |
| 不同安装路径兼容 | 搜索 direct path 和 `bin` 子目录 | 2.0.1 发布验证 | 仍需存在本机 `codex.exe` |
| 小窗数据兜底 | 后端 `last-quota.json` + 前端最后可用快照 `lastGoodQuota/displayQuota` | 类型检查通过后，便携版需实测大小窗切换 | 缓存和实时读取都不存在时仍需等待首次读取 |
| 失败态防误导 | 刷新失败后展示“暂时无法获取”和具体原因，不渲染旧账号额度 | 类型检查、Rust 编译和真实读取验证 | 适用于网络失败、账号未登录、登录态过期或账号切换后读取失败 |
| 发布包冒烟 | 启动 Portable EXE、检查窗口状态、WebView2、Codex 读取 | `scripts/verify-release.ps1` | 本机环境需具备 Codex 和 WebView2 |

## 5. 不再采用的方案

| 方案 | 不采用原因 | 替代方案 |
|---|---|---|
| 只依赖前端 localStorage 缓存额度 | 开发版重启、首次安装或缓存结构变化时不稳定 | 后端 app data 中保存 `last-quota.json` |
| 只查找固定 Codex CLI 路径 | 其他电脑可能使用 hash 子目录安装 | 扩展路径搜索并保留环境变量覆盖 |
| PowerShell 版本脚本写 UTF-8 BOM | 会让 Tauri JSON 解析失败 | UTF-8 无 BOM 写出 |
| README 展示源码结构和开发细节 | 用户下载页显得啰嗦 | README 只写用户视角，开发内容进入 `docs/` |
